1.8.4-atomic (30.03.2026)

  [Eager Loading API - with()]
*	new Cortex::with() method - fluent API for explicit eager loading
	- with(['news', 'profile']) - named relation list
	- with('news') - single string shorthand
	- with(2) - depth-based loading (all relations to N levels)
	- with(['news', 'news.tags']) - dot-notation for nested relations
	- with(null) / with(false) - clear pending eager load
	- auto-clears after find() / load() to prevent accidental reuse
*	with() + find() - preloads declared relations on the entire collection
	- leverages existing smart-loading (1 query per relation level)
	- nested relations use mega-collection pattern for batch efficiency
*	with() + load() - creates mini-collection for single-record preloading
	- enables smart-loading on records loaded via load() (not just find())
*	resolveEagerLoading() - entry point; dispatches to tree or depth mode
*	eagerLoadTree() - recursive tree walker for dot-notation paths
*	eagerLoadByDepth() - recursive depth walker; reads fieldConf for relation keys
*	supports all relation types:
	- belongs-to-one, has-one, has-many, belongs-to-many (m:m)
*	integrates with filter() - relation filters applied during eager loading

1.8.3-atomic (30.03.2026)

  [Constraint Adapter - New Module]
*	new ConstraintAdapter class (lib/db/cortex/constraintadapter.php)
	- adapter layer for FK constraints and UNIQUE composite indexes
	- future-proof design: checks for native schema-builder FK support first,
	  falls back to raw SQL per dialect (MySQL, PostgreSQL, SQLite, MSSQL)
	- when schema-builder is forked and updated with native addForeignKey(),
	  the adapter will automatically use the native method

  [FK Constraints]
*	enableForeignKeys() - sets PRAGMA foreign_keys=ON for SQLite per connection
*	addForeignKey() - ALTER TABLE based FK for MySQL/PostgreSQL/MSSQL
	- returns false on SQLite (ALTER TABLE ADD CONSTRAINT not supported)
*	foreignKeyClause() - generates inline FK SQL fragment for CREATE TABLE
*	createPivotTable() - creates m:m pivot table with inline FK (SQLite)
	or schema-builder + ALTER TABLE (other engines)
*	recreatePivotWithFK() - adds FK to existing pivot tables
	- SQLite: recreates table via temp table + data copy + rename pattern
	- non-SQLite: ALTER TABLE ADD CONSTRAINT
*	addBelongsToForeignKey() - FK for belongs-to-one relations (SET NULL default)
*	hasForeignKey() - introspection via PRAGMA foreign_key_list (SQLite),
	INFORMATION_SCHEMA (MySQL/PostgreSQL)

  [UNIQUE Composite Indexes]
*	addUniqueComposite() - uses schema-builder addIndex with unique=true
*	hasIndex() - introspection of composite indexes via PRAGMA index_list/index_info
	(SQLite), SHOW INDEX (MySQL)
*	all m:m pivot tables now get UNIQUE composite index on the two FK columns
	- prevents duplicate relation entries at the database level

  [Schema Builder Integration]
*	setup() refactored with deferred pivot creation
	- pivot tables created after main table build (not during field loop)
	- enables FK when both referenced tables exist at setup time
	- belongs-to-one FK added after table build (non-SQLite)
*	new setupForeignKeys() static method - two-pass FK approach
	- call after all related models have been setup()
	- handles m:m pivot FK ordering (chicken-and-egg problem)
	- handles belongs-to-one FK for cross-model references
	- idempotent - checks hasForeignKey() before adding

  [Test Suite - 293 Tests]
*	added Test_Constraints class (24 tests) covering:
	- enableForeignKeys() PRAGMA verification
	- createPivotTable() with UNIQUE + FK

	- UNIQUE constraint rejects duplicate pivot entries

1.8.2-atomic (29.03.2026)

  [Transactions]
*	insert(), update(), erase() now wrapped in implicit SQL transactions
	- ensures atomicity of mapper operation + m:m pivot table updates (_aftersave)
	- erase() wraps m:m reference cleanup + mapper->erase() in single transaction
	- automatic rollback on any exception during transactional block
	- exceptions are re-thrown after rollback for caller handling
*	transaction nesting support - respects user-managed outer transactions
	- checks $db->trans() before begin() to avoid double-begin
	- if outer transaction is active, Cortex skips begin/commit/rollback
	- allows multiple Cortex operations in a single user-controlled transaction
*	non-SQL engines (Jig, Mongo) are unaffected - transaction logic only activates
	when dbsType === 'sql'

  [Test Suite - 269 Tests]
*	added Test_Transaction class (22 tests) covering:
	- implicit transaction commit on insert/update/erase
	- rollback on _aftersave exception (insert and update paths)
	- user-managed outer transaction nesting (no double-begin)
	- user rollback reverts nested Cortex saves
	- multiple saves in single outer transaction
	- erase rollback on exception
	- beforeinsert event returning false prevents transaction start
